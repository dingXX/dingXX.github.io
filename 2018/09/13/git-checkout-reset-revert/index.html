平时写代码的时候，经常会遇到以下的情景

  1. 修改了一大堆内容，然后都不想要了，想变回初始状态（还没有commit到本地仓库）
  2. 一不小心将不想要的代码commit到本地仓库了，但幸好还没有push到远程仓库
  3. 将代码push到远程仓库后，发现代码有问题，需要还原为原来正常的情况

对于以上这些情况，需要我们对代码进行重置，回到某种状态。而一些git命令（如git checkout、git reset、git revert）都能很好的帮我们对代码进行重置。

但对于小编来说，经常搞不清这三个命令之间关系，也不太清楚对应命令执行后的效果是啥，导致不能在合适的时候使用合适的git命令，所以这里想好好梳理一下这三个命令操作的表现~

git项目工作区与文件状态

git 项目中有三个工作区域：git仓库、工作目录、暂存区域

  • git仓库： 文件已经被commit后，会更新到git仓库中
  • 工作目录: 只当前打开文件看到的样子
  • 暂存区域： 文件被git add了，但没有git commit（预期下次提交的样子）

文件也有对应的状态

  • 未跟踪（untracked）: 文件不在git仓库中或暂存区域中（新建的文件）
  • 未修改(unmodified)/已提交（committed）: 文件存在git仓库中的样子
  • 已修改（modified）: 文件在工作目录中被修改过的样子
  • 已暂存（staged）: 文件在暂存区的样子

提交层面的重置

这里主要是指将git项目中的工作目录变成某个提交记录时的快照样子

注意：

  • 提交层面是指所有文件都回到对应提交记录时的样子
  • 这里说的撤销不代表说将提交历史的回退，只是保证当前工作目录的文件是预期的样子

git checkout、git reset、git revert三者都能实现回滚到对应的提交记录时的样子

命令 作用 细节 是否重写历史
git checkout [commit] 将HEAD指针移到对应的commit中 此时分支没有移动，导致HEAD处于分离状态（HEAD没有被任何分支引用） 否
git reset [commit] 将HEAD和当前分支指针移动到对应的commit中 当前还是处于分支中的 是
git revert [commit] 将[commit]的修改撤销，回到[commit]的父提交时的样子 在原提交历史的基础上，新增一个还原的commit 否

已修改状态下的重置

场景1

在[commit1]下的文件状态为A1{已提交}，对某些文件做了修改，当前文件内容为A2，但没有做git add操作，（即git仓库为A1,工作目录为A2,暂存区为A1）），现在想回到A1状态

已暂存状态下的重置

场景1

在[commit1]下的文件状态为A1{已提交}，对某些文件做了修改，当前文件内容为A2，然后做了git add操作后（即git仓库为A1,工作目录为A2,暂存区为A2），想回到A1状态

场景2

在[commit1]下的文件状态为A1{已提交}，对某些文件做了修改，当前文件内容为A2，然后做了git add操作后，又对文件进行了修改，内容为A3 （即git仓库为A1,工作目录为A3,暂存区为A2），想将所有都变回A1

已提交状态下的重置

场景1

在[commit1]下的文件状态为A1{已提交},对某些文件做了修改后，执行git add和git commit，将修改内容提交，生成新的[commit2],文件状态为A2{已提交}（即git仓库为A2,工作目录为A2,暂存区为A2），想将所有的都变回A1

文件层面的重置

git checkout和git reset命令接受文件路径作为参数，对单个文件进行操作。但git revert只能对提交操作

文件已修改状态下的重置

场景1

在[commit1]的提交下，文件A的内容为A1，然后对文件A进行修改，变为A2，（A文件的状态：git仓库为A1,工作目录为A2,暂存区为A1），需要变回A1

文件已暂存状态下的重置

场景

在[commit1]的提交下，文件A的内容为A1，然后对文件A进行修改，变为A2，并将修改暂存了（git add）（A文件的状态：git仓库为A1,工作目录为A2,暂存区为A2），需要变回A1

文件已提交状态下的重置

场景

在[commit1]的提交下，文件A的内容为A1，然后对文件A进行修改，变为A2，并将修改提交了[commit2]（A文件的状态：git仓库为A2,工作目录为A2,暂存区为A2;HEAD指向[commit2]）
