在写修bug的时候，有时会毫无头绪，完全不知哪出了问题，这时候的git某些命令就发挥用处了。以下主要介绍几种不太常用（对于我来说），但适合用于定位问题的几个git命令，以作备忘。

git bisect

如果一个bug，你能快速定位问题，那一定是个好bug。但实际上总存在一些陈年老bug，你不知道她从何而来，从何时存在，只是某一天被某人无意中发现，从此成为你的噩梦…

那git bisect就是个神器，至少它可以快速地告诉你，bug从何时存在

git bisect命令——二分查找，它会对提交历史进行二分查找以定位哪个提交引进了问题。

相关命令操作

步骤1 启动二分查找，且定义坏的提交和好的提交

注意事项

  1. [good_commit]一定要是[bad_commit]的祖先
  2. 执行git bisect时所在的目录，一定要是git的根目录，处于子目录执行命令会报错

步骤2 判断中间提交是好提交还是坏提交

执行完步骤1后，
git 会输入以下信息，告知整个过程大概需要几步roughly 1 step，还有几个提交版本需要测试1 revision left，且告知中间提交（[bad_commit]和[good_commit]的中间一个提交）的版本号，并将工作区域切换到中间提交的状态

此时，我们需要判断这个中间提交是一个好的提交（bug不存在了）还是一个坏的提交（bug依然存在），确定好之后，告诉git 结果

步骤3 确定bug的源头提交

循环执行步骤2之后，git 最后会告知哪个提交分支是the first bad commit，并展示出该提交的改动。这时候我们就能通过改动大概定位bug是怎么来的啦~

步骤4 结束整个二分查找

定位好问题后，我们需要将HEAD指针重置到最开始的位置，结束整个二分查找。以便我们修改bug。

查看之git bisect log

在整个二分查找过程中（git bisect start 和git bisect reset之间），我们都可以执行git bisect log 查看目前已经检查过的提交版本，以及他们的状态（good or bad）

带脚本自动化测试

我们也可以编写一个自动化测试代码，当结果返回0时，则认为是好提交，非0则为坏提交

这会一次性跑完，直接告知第一个坏提交的版本号

git blame

git blame [fileName]可以查看某个文件的某行代码是谁写的（找锅主）

如果只想看某几行的内容，可通过-L属性指定行数
