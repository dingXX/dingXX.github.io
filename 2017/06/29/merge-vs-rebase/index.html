merge-vs-rebase

Jun 29, 2017 | 技术 | 阅读

刚开始用git客户端smartgit时，当对新加的仓库进行同名分支的pull操作的时候，总会要选择是merge还是rebase。刚开始的使用不懂，别人说要选择rebase就选了。但真正用起来，发现有冲突时，就一脸懵逼了（主要是学艺不精）。之后，在别人的讲解下，终于稍微了解了两者的区别。

what？

没有论点的结论：merge模式和rebase模式，两者的目的都是为了将一个分支里的更改并入到另一个分支中。只是实现的方式不一样罢了。当然最后呈现的东西也不太一样。

原始的：

在dev分支下，执行git rebase master后的dev分支：

merge

原理： 将两个分支的最后一个共同节点（C1）以及两个分支的最后一个节点(C3，C5)进行三者合并，生成一个新的节点(C6),提交到被合并的分支（dev）中。
特点： 生成一个新节点。不改变分支的项目历史。

rebase

原理：先将被合并分支(dev)回退到两个分支的最后一个共同节点(C1),再将合并分支（master）上的节点并入过来。再将被合并分支(dev)的节点一个一个地并入（解决冲突）。
特点：不生成新节点。改变分支的项目历史。得到一个清晰的历史记录。

how？

刚开始用git的使用，最怕的事情就是冲突（现在也怕）。

用merge模式时的冲突解决

merge模式下的合并其实只发生一次，所以出现冲突也可能只出现一次。

  1. 出现冲突

  2. 解决冲突

  3. git add .暂存冲突的文件

  4. git commit -m 'merge file'提交解决后的冲突文件

用rebase模式时的冲突解决

rebase模式下的冲突是发生在将被合并分支(dev)的节点一个一个地并入时，所以这个冲突可能要解决多次（<=dev分支的自己的节点的个数）。
情景模拟：
在上面的栗子里，用rebase模式时，dev分支“恢复”到了C1,再合入了master的C2和C3.(到目前为止是不会有冲突出现的)。这时，

  1. 需要将C4合进去，发现有冲突了，操作会暂停，并告诉你哪个文件有冲突。

  2. 然后，我们修改了文件的冲突了。
    就对冲突的文件进行git add .操作。（暂存解决冲突后的文件）

  3. 执行git rebase --continue

  4. 重复第一步，将C5合进来。（如果那个节点没有冲突，会直接合进来，再合下一个节点）

git rebase --abort 放弃rebase。（在rebase过程中，只要执行了这个命令，之前做的rebase操作都会被取消，还原执行git rebase master前的状态）

rebase黄金法则

感觉看到的文章和自己试验的不一样。。。。
之前看到的很多关于rebase黄金法则的介绍时，都说到的一句就是“不要在公共的分支使用rebase”。其实自己一直都不太能理解这句话的。自我理解的意思是“不要改变公共分支的提交历史”
之后自己做了一下实验，发现也没有别人说的混乱问题，所以还是不太懂这黄金法则…

以下是我之前做的实验。

  1. 一个应该是能被接受的rebase(同名分支的合并用rebase)
    栗子：
    本地分支(dev)rebase远程分支(origin/dev)后，将devpush到origin/dev

这种情况下其实是将本地dev的新提交的节点的基底改为origin/dev的最新节点。之后执行push操作时，并不会改变origin/dev的提交历史。
原始提交记录：

git rebase origin/dev后

git push origin dev后

  1. 对本地公共分支的合并用rebase
    栗子：
    将本地分支(dev)rebase对本地的公共分支(master) ，再将本地分支(master)合并(merge或rebase)到远程的master分支。查看远程master分支是否有提交历史的改变。
    2.1 将本地分支(dev)rebase对本地的公共分支(master)

原始状态：

在master分支中，git rebase dev

2.2.1 将远程的master分支rebase到本地的master分支
