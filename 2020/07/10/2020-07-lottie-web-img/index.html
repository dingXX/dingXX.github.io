lottie-web的动画图片加载处理

Jul 10, 2020 | 阅读

好久之前做了个兑换页面，里面有个开奖动画，设计师给的是lottie的动画，按官网的写法，引入了对应的json文件，却发现动画执行不了（图片是裂图），但打开设计师给的文件中的demo.html却能正常展示动画的

背景

  • js 代码引入对应的json文件

但展示出来的效果确实这样的…

这一看就是图片没加载成功呀，但为啥图片没有加载成功了，我又尝试引用了官网线上展示的动画json，却发现是可以成功展示动画的…

找出问题

然后我就看了一下设计师提供的json文件的格式，会发现它定义个assets数组，里面包含了一些资源的定义

看这结构应该能推断出，这些资源的加载应该是根据u(文件加名)和p(文件名)加载的，无码无真相，只能去看lottie代码是怎么加载资源的，找到lottie-web入口文件，用assets作为关键字搜索，找到了以下这段获取资源路径的代码

再想想项目是用webpack打包的，打包出来的图片位置和文件名了，按json文件的定义（相对路径，走到了else部分），那一定是找不到对应的文件的呀，所以才导致了出现裂图的情况。

解决

那有什么办法可以让json中的图片路径是一个正常能访问的路径呢？？（绝对路径 or base64）

base64

第一反应就是直接设计师解决，如果他们生成这个动画时用的图片是 base64 的话，那项目只要维护这一份json就好了，如官网提供的部分lottie动画

然后就找了一下万能的网络，搜到了这篇文章 动画落地 | 从 AE 制作到落地开发

Lottie 早期的版本是不支持图片类动画的，导出 json 之后会自动生成一个 img 的资源文件夹，播放 .json 文件时，需要解压资源压缩包到本地目录才能正常播放。从 bodymovin V 5.1.15 之后，Lottie 将图片转为 base 64 编码，使用字符代替图像地址，并封装在 json 里，直接播放一个 .json 文件，不用再拖着一个资源文件夹了。

兴致匆匆的将文章丢给了设计童鞋，让他们重新导出个带base64图片的json 文件，但… 被设计童鞋拒绝呢，理由是直接将图片变成base64的话，图片无法压缩，最后得到的json文件大小是10m 多（38个图片）

优点

  • 开发侧无需维护图片资源，只要维护一份json就好，感觉适用于多客户端使用同一份动画的情况

缺点

  • 图片转base64 会导致 json文件大小增大
  • 当其中一张图片改变时，会修改整个文件

绝对路径

好吧，那只能换个思路，那就再尝试一下将资源路径改成 绝对路径

json 也可以理解成是一个js对象呀，那直接对json中的assets对象进行处理，将图片对象变成一个绝对路径就可以吧

优点

  • 可支持图片多语言
  • 单张图片改动不会引起json文件的改动

缺点

  • 开发侧需要维护图片资源

多语言处理

由于我们的业务是需要支持多语言的，如果动画里存在文字的话，那就需要对动画考虑多语言问题了

图片文字

如果文字是图片形式的话，那可以考虑在处理成绝对路径时，做一个语言判断，替换文字图片的文件名

总结

本文总结了lottie_web的图片加载过程，提供了项目中使用lottie_web时的图片资源加载不正确问题

  • 设计师在导出json文件时，可以选择将图片转换为base64
  • 开发侧在项目中，用js修改json文件中的assets对象的图片路径
