之前接到一个需求，是由各种卡列表页、卡详情页和卡消费页组成。
其中，会存在以下一些交互

  1. 卡列表页需要滑到底部自动刷新
  2. 从卡列表页/卡消费页点击跳去卡详情页，卡详情页只有去卡消耗页的出口，要回到列表页只能后退返回
  3. 各列表页（有效卡列表/已消耗卡列表等）间会相互跳转
  4. 卡消耗页需要选择部分卡后，才能消耗；消耗完后，会影响其他列表页的数据

根据上面的要求，觉得使用vue 的keep-alive 组件实现部分组件的缓存，在交互体验上应该有较大的提升。

  1. 从列表页跳去详情页后再回来，列表页被缓存过的话，可以回到离开前的状态，不需要再重新拉数据
  2. 在卡消耗页有部分卡被选择时，再点击去卡详情页回来后，使用缓存的话，离开前的选择记录可以保留

分析

根据以上交互，我们需要先确定一下哪些页面需要被缓存

  1. 卡列表页/卡消耗页需要被缓存
  2. 卡详情页不需要被缓存(每次的卡id不一样，都需要重新拉数据，没有缓存的必要)
  3. 在卡消耗页里，卡被消耗后，所有的列表页应该被更新

实现

使用meta.keepAlive控制是否缓存

先搜了一下大众写法
原理

  1. 通过各个路由的keepAlive属性，确定路由页面是否放在缓存组件中，已实现部分路由组件缓存，部分不缓存
  2. 在beforeEach根据to路由中定义的keepAliveExcludeFrom属性，决定当前是否需要keepAlive

但这么实现后，发现路由间的跳转后，缓存不符合预期
如：进入列表页–>去详情页–>去卡消耗页(并消耗卡)–>后退返回到列表页（此时的列表页是缓存的数据）【但实际上想要的是重新拉取列表页数据】

重新思考后，发现上述的方案并不能满足这个需求。
该需求的要求是当卡被消耗后，所有路由组件的缓存都是无效了的
但上述方法只能保证当from页面是卡消耗页时，当前页面才不使用缓存（但实际上该页面的缓存组件依然存在）

使用keep-alive的include属性

查看官网后，发现keep-alive 有个include属性，允许组件有条件地缓存，而且可以动态使用。

大概看了一下keep-alive的实现，其会watch include和exclude的值，当不符合include要求时，会destroy掉这个组件的缓存

这么看，好像符合这个需求的要求，只要在卡消耗后，将include设为空，触发所有缓存组件的销毁，过段时间后再加回去，在需要缓存的时候，继续起效。

在卡消费后，触发removeKeepAlive事件

使用keep-alive注意事项

  1. keep-alive只对直属子组件有效，即如果使用嵌套路由时，子路由是不会直接被缓存的
  2. keep-alive中的include匹配的值是组件的name，不是路由的name！！！
